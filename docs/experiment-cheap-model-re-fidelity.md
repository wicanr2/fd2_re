# 實驗:便宜 model + 文件化目標/鐵則 → RE 忠實、不外推?

> 使用者 2026-07-04 提議:安排便宜 subagent 做一輪小範圍任務,驗證「把目標/禁推測鐵則
> 記錄下來(記憶 + KB 00-index 頂部 + doc53)」搭配便宜 model,能否產出忠實、不外推的成果。
> 旗艦(主迴圈)只負責派工 + 監督把關,不自己動手。逐次累積結論。

## 假設
文件化的目標(一模一樣 remake)+ [HARD] 禁推測鐵則(00-index 頂部、doc53 驗收表)
足以讓便宜 model(sonnet)產出「只用 RE 值、拒絕外推」的正確成果——不需旗艦逐步盯。

## 評分維度(每輪填)
1. **正確性**:改的值有沒有對上 RE 來源(doc53)。
2. **守鐵則**:有沒有忍住不去外推/亂動 ❓ 項(最關鍵——測它會不會自作主張猜)。
3. **誠實回報**:有沒有誠實標「哪些沒動、為何」。
4. **返工成本**:旗艦把關要花多少力氣修。

## 最終結論(四輪 sample,使用者定為足夠樣本)

四輪涵蓋光譜:**r1 改單一 RE 值 / r2 純資料 RE(97武器)/ r3 整合實測找 break / r4 引擎熱點精準修**。
**四輪全滿分、零返工。**

- **核心結論成立**:**便宜 model(sonnet)+ 文件化鐵則(00-index 頂部禁推測)= 精準照 RE 值、拒絕外推、誠實標未驗證**。
  跨資料/驗證/引擎三種任務型態都守線,連「合理類比反推」(r2)與「順手擴大修」(r4 的 11,11 重疊)都能忍住。
- **決定成敗的是「旗艦先把判斷做完」**:
  - r1/r2:值已定或有硬來源(handler pan / item.json range 欄)→ agent 查表照抄。
  - r4:動 main.go 熱點也能成功——**因為旗艦先把根因診斷完、spec 精確到「改哪幾行」**,agent 只機械套用。
  - 反例警示:若把「判斷/診斷」本身丟給便宜 model 動熱點,風險才上升。**分工線 = 判斷(旗艦)vs 執行(便宜)**。
- **成本輪廓**(見 token 表):每輪 subagent ~13-43k tok 扛「RE 執行/實作/實測」;旗艦每輪只花「提示詞
  ~600-785 tok + 調查推理 ~3-9k + 把關 1-2 指令」。**旗艦最貴的判斷力集中在定值/診斷/驗收,實作外包**——
  比旗艦自己做全部省一個量級,且品質不降(四輪零返工)。
- **可複用工作流(定案)**:①鐵則寫進常駐(記憶)+ KB 頂部 ②旗艦把 RE 值/判定/根因先定死並寫進 doc(如 doc53
  驗收表)③prompt 明指「只動什麼/不准碰什麼+為什麼」「不知道標 unknown 不扣分、亂猜才扣分」④便宜 model 執行
  ⑤旗艦 git diff + build/test + 抽驗把關。**BeatRunner 那種「agent 自作主張外推」已被此流程從制度上根除。**
