## 0. 結論先講(細節見下)

| 問題 | 結論 | 信心 |
|---|---|---|
| pose 是否含相對位移(使用者最想知道) | **否**——影片全程兩角色原地不動,pose 換值期間座標不變 | **高(影片直證)** |
| pose(+3) 與 方向(+0x0B)/跑步幀(+0x0C) 是否同一欄位 | **不是**,structural 上是 80-byte 單位結構裡三個不同 offset(+3 vs +0x0B vs +0x0C),互不重疊 | 高(靜態反組譯,doc31/47 交叉核對) |
| pose 是否驅動可見的角色朝向/姿態 | **至少一例證實會**——草地幕亞雷斯的 sprite 在對話過程中從「側面(面左)」變成「正面(面下)」,與 acting 資料的 pose 變化時間窗吻合 | 中(影片證實有變化,但幀取樣 0.5s 粒度,無法逐 tick 精確對應) |
| pose 0/1/2/3 個別視覺意義 | **未解**——嘗試套用 doc31 的方向枚舉(0下1左2上3右)會與觀察到的時序矛盾,不強行採用 | 未解,誠實標註 |
| acting `unit` 索引 → 角色 | **只確認相對順序**(先出現=亞雷斯、後出現=坐地友人),對應到 FDFIELD 30-unit roster 的**絕對槽號矛盾**(見 §4) | 中(順序高信心,絕對槽號未解) |

## 1. 方法

1. 用 doc47 §3 beat 表鎖定 acting id 對應的真實劇情時刻:
   - `ACT(0x63)` = 索爾王座廳走入(t≈36–43s,camera-follow 已由 doc47 §9 定案,本輪只做交叉驗證)
   - `ACT(0x65)`–`ACT(0x69)` = 草地「亞雷斯撞見友人」(t≈128–144s)——使用者/team-lead 建議的「乾淨、單位少」樣本
2. `ffmpeg` 依 doc46 時間軸抽幀(2–10 fps 視需求,對話框出現時角色常被遮一半,须挑對 crop 區域)
3. 對照 `tools/decode_story_text.py extracted/raw/FDTXT/FDTXT_033.bin`(逐句劇本,含說話者標籤)
   確認畫面上「誰在講話」,與 acting 幀資料的座標/時序交叉比對
4. 有影格證據才寫進結論表;純推理不佐證的一律標「未解」

## 2. 王座廳走入(ACT 0x63)——複驗 doc47 結論,無新資訊但確認一致

`ffmpeg -ss 34 -to 46 -vf fps=2` 抽幀(24 張,t=34.0–46.0s)清楚看到:
索爾(藍髮小人)從紅毯底端一路走到頂端王座區,鏡頭同步上移(與 doc47 §9 的「camY 30→26 同步索爾 Y −4」
完全吻合),抵達後停下、對話框浮現「兒臣索爾，晉見父王陛下。」

`acting_decoded_throne.txt` 裡 `id=0x63` 全程只有 2 幀,**pose 值全程=0,沒有變化**(`frame[0]`
`frame[1]` 皆 `unit=61,pose=0`,只有拍數 5→4)。這與 doc47 §9 的結論一致:此 acting 呼叫的
`(unit,pose)` 資料本身是殘留/節拍器,**真正的走位由引擎逐 tick 步進單位 2(索爾)的座標驅動**,
跟這筆 acting 寫的 `unit=61`(超出 21 個有效槽)無關。本輪影片複驗**沒有推翻**這個結論,
只是確認「pose 全程不變」與「影片裡索爾確實只是直線走位、無轉身/手勢」對得上。

## 3. 草地「亞雷斯撞見友人」(ACT 0x65–0x69)——新分析

### 3.1 畫面內容(影格證據)

`ffmpeg -ss 128 -to 144.5` + 定位 crop(`crop=850:170:400:440`,螢幕座標經
`full_check.png` 逐步校正,見過程記錄)清楚拍到兩個角色:

- **左側(x≈470–550)**:紅衣友人,坐姿/面朝下,**全程(t=128.0–142.5s)姿態肉眼看不出變化**。
- **右側(x≈1120–1200)**:亞雷斯(藍衣、帽子),對話框портrait 確認「亞雷斯」開場白
  「哈!找你找了半天,居然是躲在這裡打瞌睡」——他先以「側面朝左」姿態站立(t≈129–133s,
  對話框剛開始出現、亞雷斯剛走到定位時),到 t≈139.0s(友人「唉!」嘆氣那個對話空檔,
  地圖完整可見)時已變成「正面朝下」姿態。**這是本輪影片能直接驗證到的唯一一次 pose 造成
  可見姿態改變**。

### 3.2 座標:全程未變(關鍵結論)

用 0.1–0.5s 間隔連續抽幀比對兩角色像素座標,**從 t=128.0 到 t=142.5s(覆蓋 acting id 0x65 到
0x69 全部呼叫),兩角色的螢幕座標沒有偵測到位移**——只有亞雷斯的「面向」變了,人沒有移動。
**直接回答使用者的問題:pose 不是相對步進位移,是原地的姿態/朝向切換。**

### 3.3 pose 值本身:對不上單一乾淨規則,誠實標未解

`acting_decoded_throne.txt` 裡與此幕相關的 id:

```
0x65: 16 units(idx 0-15) 全部 pose=2,單幀(建立鏡頭/整批單位歸位,疑為離開王座廳的收尾,非草地本身可見內容)
0x66: unit16,17 兩者同步:(2,2)→(0,0)→(2,2)→(3,3)→(1,1)→(1,1)  6幀,每幀8拍(末幀0拍)
0x67: unit16,17 (3,3) 單幀,15拍(長停留)
0x68: unit1 單獨變化:3→2→3(bit7,4拍)
0x69: unit16 單獨 pose=3(8拍,bit7)
```

嘗試套用 doc31 已驗證的方向枚舉(`0=下 1=左 2=上 3=右`)去對 0x66 的序列,和影片實測的
「亞雷斯側面朝左(t129-133)→正面朝下(t139)」時序做交叉比對:**若 pose 直接複用方向枚舉,
0x66 序列結尾應停在 `pose=0`(下)才能對上 t139 的正面觀察,但實際序列結尾是 `pose=1`(左)**,
兩者矛盾。**不採信「pose = 方向枚舉」這個假說**,也不編一個新枚舉去湊合——按規則 64,
沒有幀級證據支持的對應關係一律標「未解」。

可能原因(均未驗證,列為待查方向,不當結論用):
- 抽樣頻率(0.5s)遠粗於遊戲 tick(8 拍可能 <0.5s 內完成),漏看中間幀,誤把「頭尾」當「全部」比對;
- pose 不是逐字複用方向枚舉,而是另一張「劇情姿態」表(待機/驚訝/手勢/僵住等),數值恰好也是 0-3
  純屬欄位寬度巧合;
- `0x66`/`0x67`/`0x69` 動的是 unit16/17,`0x68` 動的是 unit1——**同一幕內 unit 編號不連續**,
  意味著這幾筆 acting 呼叫可能各自對應「螢幕上不同的兩人組」或有我們沒抓到的
  index 重新指派,需要 dosbox 斷點才能坐實,不能靠影片肉眼分辨這麼小的 sprite 差異。

### 3.4 pose vs 方向(+0x0B)vs 跑步動作幀(+0x0C):結構上確認互不相同

doc31 §6 反組譯戰場單位繪製碼(`0x12823`–`0x12932`)讀的是 `unit[+0x0A]`(sprite組)、
`unit[+0x0B]`(方向)、`unit[+0x26]`(動作幀offset)——**這條繪製路徑完全沒有讀 `unit[+3]`**。
也就是說:**doc47 的 acting pose(+3)不是戰鬥/戰場標準 FDICON 繪製碼的輸入**,它是
另一條路徑(doc47 §2:acting 每幀呼叫 `0x11cac`/`0x11eee` 重繪)在用的欄位。

結構性結論(高信心,純位址比對,不靠影片):
- **pose(+3)、方向(+0x0B)、跑步動作幀(+0x0C)是同一個 80-byte 單位結構裡三個不同 byte**,
  不重疊、不是別名。
- **tick(+4)** 的角色可以從 doc47 的「移動步進」推廣:本輪看到 pose 純姿態切換(無位移)時,
  tick 一樣是「這一幀要保持幾個 engine tick 才推進下一幀」的倒數計時器——**tick 是通用的
  「保持幀時長」倒數,不是專屬走路步進**;pose 才是「這一 tick 要呈現什麼視覺狀態」,
  兩者分工在移動與純姿態場景下是一致的(移動時 pose 意義上等同一種特殊狀態驅動位置步進,
  純對話時 pose 只換姿態不挪位置)。
- **`0x11cac`(acting 用的重繪)大概率是與 doc31 戰鬥 FDICON 繪製碼(`0x128e0`)不同的另一條
  繪製路徑**(可能是通用「地圖/劇情模式」繪製,battle 專屬繪製碼另外走),這能解釋為何
  pose 不必經過 `+0x0B`/`+0x0C` 也能改變畫面——**此點為推論,未反組譯 `0x11cac` 內部讀了
  哪個 offset 來坐實**,列入待辦。

## 4. acting unit-index → 角色:順序可信,絕對槽號矛盾(未解)

**相對順序高信心**:0x66/0x67/0x69 涉及的兩個 unit(16,17)在畫面上就是「亞雷斯 + 坐地友人」
這兩人(順序:亞雷斯先出現且先開口,對話框肖像確認)。

**絕對槽號矛盾**:用 `tools/parse_field.py extracted/raw 32` 印出的 FDFIELD map32 roster,
槽 16/17 是兩名 `portrait=69` 的守衛(與 doc47 §8 槽序表一致:槽13-20=守衛69×8)——
但影片裡這兩個位置明明是「亞雷斯(綠帽紅衣)」與「坐地友人(紅衣)」,不是兩個一模一樣的守衛。
**沒有在本輪坐實原因**,但有一條有旁證支持、尚未反組譯驗證的假說:

- `docs/knowledge-base/40-speaker-portrait-mapping.md` 已證實:`[0x53A45]`(戰場單位表)
  **是隨地圖/場景重新填入的資料**,不是寫死表;同一數字（陣列 index 或身分標籤）在不同時刻
  可以指向完全不同的單位。doc47 Part2(密林幕)beat 表也看到 `0x10b4e(群組)`(spawn)呼叫,
  代表 acting 播放前後會有 spawn 動作重填這張表。
- **合理推測**:草地幕在 `ACT(0x65)`(units 0-15 全設 pose2)前後,很可能有一次「重新填充」
  (可能是被摺進 doc47 §3 beat 表沒展開的 spawn/init 呼叫),把槽 16/17 從「王座廳戰鬥用守衛69」
  改填成「亞雷斯 + 友人」的劇情演員身分,供這段 cutscene 專用——**同一張表在不同時刻被
  拿來做不同用途(戰鬥 roster vs 劇情演員表)**,而非兩套系統各自維護一張表。
- 這只是**未經 dosbox 記憶體 dump 驗證的假說**,不當結論寫。要坐實需要:草地幕開演瞬間
  斷點讀 `[0x24B2F0]`(doc47 §8 給的執行期位址)+idx16/17 的 `byte[+7]`(肖像)欄位,
  看是否確實已從 69 改寫成亞雷斯/友人的肖像 id。**列入下一輪待辦,本輪不強行下結論。**

## 5. 對 remake 的直接可用結論

1. **cutscene 角色的「原地換姿態」是安全可抄的機制**:remake 可以直接做「站樁播放一小段
   姿態序列而不移動座標」的動畫,不需要額外設計位移插值——原版影片證實這條路徑本來就是
   純姿態切換。
2. **不要假設 pose 直接映射現有的 4 方向枚舉**——目前證據不支持,remake 若要照抄 pose
   序列,應該當成獨立的「表演姿態」索引(配 FIGANI 或專用 cutscene sprite),不要重用
   FDICON 的方向欄位語意,避免把不確定的假說寫死進資料驅動系統。
3. **acting 資料的 unit 索引在 remake 端不必强行對回 FDFIELD 槽號**——remake 的過場系統
   本來就可以用「劇情本地演員表」(doc50 cutscene-script-system-design 應該已經是這個設計),
   這正好呼應本輪發現的「原版本身可能也是這樣做的」,不是 remake 偷懶,是原版真的这样设计。
