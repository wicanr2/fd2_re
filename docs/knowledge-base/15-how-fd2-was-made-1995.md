# 15 — 1995 年,他們怎麼做出《炎龍騎士團2》

> 這是一篇總覽:把本知識庫各篇拆解的技術,綜合成「一支 1995 年的台灣團隊,在 DOS 上怎麼把這款戰棋
> RPG 做出來」的完整紀錄。給後人看——當年沒有現成引擎、沒有寬裕記憶體、沒有中文系統,他們用一套
> 務實到近乎土法煉鋼、卻處處藏巧思的做法,把一個世界塞進幾 MB 的檔案裡。各技術細節見對應章節編號。

## 時代背景

- **遊戲**:《炎龍騎士團2》(Flame Dragon 2 — Legend of Golden Castle),漢堂國際,1995。
- **平台**:MS-DOS,VGA 320×200 256 色(mode 13h),靠 DOS extender 進 32-bit 保護模式。
- **限制**:記憶體以 MB 計、無硬體加速、DOS 不支援中文、磁碟空間珍貴。**一切都要自己造,且要省。**

## 一、工具鏈:把現成的商用件接起來(見 `04`)

他們沒有自己寫編譯器或音效卡驅動,而是把當年的商用工具組起來:

- **Watcom C/C++ 32-bit**(1988–1993 版)編譯,產出 **LE(Linear Executable)**。遊戲邏輯是 32-bit C。
- **Rational DOS/4GW** 當 DOS extender,把程式帶進 32-bit 平坦記憶體(才能定址大量資料)。
- **Miles Design Audio Interface Library (AIL) V3.02** 處理音效——一份音樂資料能在 AdLib/SB/MPU-401/GUS
  等各種音效卡上播放(故隨遊戲附一整套 `.MDI`/`.DIG` 驅動)。
- **自製動畫工具 AFM(Animation File Manager)v1.00**,作者署名 **Lo Yuan Tsung,1993**——
  程式留在 `ANI.DAT` 裡的版權橫幅,把當年工程師的名字一起保存了下來。

> 心法:**能買的就接,核心的才自造**。省下的力氣拿去做內容。

## 二、資料架構:一種容器吃天下(見 `01`)

幾乎所有資產——地圖、圖、動畫、文字、音樂、頭像——都打包成**同一種極簡容器**:
`"LLLLLL"` magic + 一張 uint32 offset 目錄。判別與解包只需「讀第一個 offset 回推項數」。

```
LLLLLL | offset[0] offset[1] … | 資源0 | 資源1 | …
```

**自描述**是反覆出現的設計慣性:目錄項數 = (第一個 offset − 6) / 4;動畫的幀數 = (第一個幀 offset − 8) / 4。
連音樂、動畫之間都用同樣的「3-byte 分隔標記」切組。**一套思路套用到底**,寫一支解包器就能拆全部。

## 三、畫面:VGA 256 色 + 多種量身 RLE(見 `05`/`06`/`01`§7)

- **基準**:320×200、8-bit 調色盤索引;調色盤是 `FDOTHER` 資源 #0(256×RGB,6-bit,×4 轉 8-bit)。
- **壓縮**:不同用途配不同 RLE,各自最省:
  - **背景 / 標題**(矩形實心):`c≥0x80` literal、`c<0x80` run。
  - **戰鬥動畫 sprite**(非矩形,要透明):4 模式 RLE(色彩 run / dither 陰影 / literal / 透明 skip),
    控制 byte 高 2 bit 選模式、低 6 bit 給長度。
  - **人物頭像**(矩形):更簡的「`b≤0xC0` 字面 / `b>0xC0` run」。
- **基本圖塊 24×24**;戰鬥背景 320×100;頭像 80×80;標題 320×200。
- 「正確解碼必輸出剛好 W×H」成了我們反推這些格式的決定性判據。

## 四、動畫:自製 AFM + 幀封裝(見 `06`)

戰鬥動畫(`FIGANI`,15 MB、264 段、2118 幀)是全作最大宗。每段動畫自描述幀數 + uint32 幀 offset 表;
每幀 13-byte 標頭(外框寬高、真實點陣寬高在 +9/+11)後接 sprite RLE。
解出來是一個個藍灰盔甲、紅披風的騎士揮劍動作,連地面陰影都用 dither 模式畫。

## 五、中文:沒有中文系統,就自帶字型 + 內部索引(見 `08`/`14`)

DOS 不顯示中文,他們的解法是 1990 年代台灣遊戲的通行做法,但做到極致:

- **自製 16×16 點陣字型**(`FDOTHER` 資源 #4,1824 個字模)。
- **文字不存 Big5,存字型字模的 uint16 索引**;只收錄遊戲用到的字,省空間。
- **字型是「大雜燴」**:數字 0–9、英文 A–Z、漢字(依首次出現序)、標點,**甚至把雙字元代號做成單一字模**
  ——天空之城機器人篇的型號(`73`/`C2`/`A1`/`X-`…)和兩位數(`11`/`22`)各佔一格。當年沒有彈性排版,
  就把整組符號塞成一個字。
- **對話框由控制碼驅動**(渲染器 `0x16D00`):`0xFFEF` 開框並從 `DATO.DAT` 載說話者頭像(137 角色 ×4 嘴型幀)、
  `0xFFEE/ED/EC` 開不同位置的框、`0xFFFE` 軟換行(每框 3 行)、`0xFFFD` 按鍵翻頁、`0xFFFF` 結束。
- 我們已把字模 ↔ Unicode 雙向對照建好(`glyph_map.json` / `unicode_to_glyph.json`),全 35 章劇情可讀可重打。

## 六、音樂:一份 XMIDI,各卡通吃(見 `07`/`12`)

音樂是 **XMIDI**(Miles 的 MIDI 擴充),放在 `FDMUS.DAT`(15 首)。播放走 AIL 序列 API
(`AIL_init_sequence`→`set_loop_count`→`start_sequence`)。換場景時 `play_bgm(track)`(`0x26777`)先比對
目前曲號 `[0x1A11]`(同曲不重播),`track=-1` 停曲,否則載入並播放——32 處呼叫就構成了「場景→配樂」表。

## 七、遊戲規則:資料表 + 80-byte 單位 + 評分式 AI(見 `03`/`11`/`13`/`10`)

- **數值全在 `FD2.EXE` 的資料表**:物品(23B)、法術(7B)、敵我單位(10B)、升級成長(11B)、職業抗性/暴擊…
  我們依錨定特徵定位後整批 dump,並與青衫攻略字面值交叉驗證無誤。
- **場上每單位 80 byte**:座標、圖形/方向/動作、行動狀態(`AA`:未動/死/完畢)、陣營(`BB`:敵/友/己)、
  肖像、物品 8、法術 5 組 bitfield、各屬性。
- **AI**(`0x15140`):對行動單位 flood-fill 算可達格,逐落點×逐目標評分
  (預期傷害 = 我 AP×地形% − 目標 DP×地形%;可擊殺 ×2;狀態 ×1.5),取最高分行動。敵與友軍 NPC 共用此引擎。
- **選單**:行動狀態機(全員 `AA=完畢` → 換邊)、方向鍵游標、Enter/Space 確認、ESC 取消;
  法術面板 `Get_EasyMagic`(讀 5 組 bitfield 顯示可用法術)。
- **省記憶體的巧思**:sprite **不分陣營只存一份**,繪製時用 **23 張 remap LUT**(`FDOTHER` 資源 #3,`LMI1` 容器)
  即時換色——已行動變灰、敵我染色、夜戰色調都靠查表。一份美術,多種樣貌。

## 貫穿全作的設計哲學

1. **能省則省**:只收用到的字、sprite 不分陣營靠 LUT 變色、各用途配最省的 RLE。
2. **自描述、一致**:同一種容器 + 自推項數 + 分隔標記,套用到所有資產。
3. **商用件接核心、核心才自造**:Watcom/DOS4GW/Miles 接起來,把力氣留給引擎與內容。
4. **務實的土法**:把雙字元代號塞成一個字模、用控制碼負值分派、把調色做成查表——
   都不優雅,但在當年的限制下「剛剛好夠用」,而且跑得動。

> 這些就是 1995 年一支台灣團隊做一款 DOS 戰棋 RPG 的真實樣貌:在很小的盒子裡,用很省的辦法,
> 裝下一個有劇情、有戰棋、有動畫與配樂的世界。本知識庫把這套做法逐一拆解、驗證、留存下來。
