## 1. 方法與校準(可信度基礎)

1. `ffmpeg` 從 `video/fd2-ch1.mp4` 抽幀:粗掃(2fps)先鎖定「有無對話框」時段,細抽(5–10fps)
   鎖定走位發生的窗口。
2. 螢幕內容 viewport 校準:量得該片全程黑邊內的有效內容區為 `rows[107:974] cols[17:1424]`
   (1440×1080 原始錄影裡的實際遊戲畫面範圍,黑邊另算)。
3. tile pitch:用 `scipy` 正規化互相關(zero-mean NCC via FFT)以此 viewport 對 `extracted/maps/map32.png`
   (432×1224,24px/格原生)做尺度掃描,最佳解 **pitch=108px、比對位置=viewport 左上角對齊 map32
   (col0, row1031)**,相關係數 0.37(比對地形明顯的路徑彎折形狀,非雜訊峰值)。換算:
   `tile_col = (screen_x-17)/108`,`tile_row = 42.958 + (screen_y-107)/108`。
4. **交叉驗證**:t=128.0s(幕剛淡入,尚未有任何走位)量出索爾 (4.53,46.18)、亞雷斯 (12.5–12.64,47.16–47.20)
   ——與 `campaign_full.json` 現有 `actors` 靜態值 `fig0=(4,46)`、`fig4=(13,47)` **幾乎完全吻合**,
   證實本輪座標系與既有資料同一套,可直接比較。
5. 角色追蹤:用 t=128.0 幀直接裁出索爾/亞雷斯的站姿當模板,對每個「無對話框」的影格做**全視窗**
   NCC 搜尋(非侷限鄰域搜尋——首次嘗試用「上一幀位置±130px 鄰域搜」被角色快速移動甩開、鎖死在
   地形雜訊上長達 70 秀,已捨棄改全域搜尋,見 §5 教訓)。
6. 對話框判定:取螢幕固定點 `(700,300)` 的色彩(對話框內部藍色 `[38,67,136]` vs 草地 `[~90,113,17]`)
   分類每幀為 DIALOG/MAP/BLACK,只在 MAP 幀量角色座標。

## 4. 「帶他離開」畫面沒有量到——誠實標未解/不存在

團隊描述「索爾走向亞雷斯,然後帶亞雷斯離開」的後半段(兩人一起再走一段路離開畫面),
**本輪逐幀量測沒有找到對應畫面**:索爾抵達亞雷斯身旁後 0.2–0.4 秒內螢幕就開始淡出(t≈203.0起,
連續多幀變暗直到全黑,約 1.5 秒淡出時長,對應 doc46 §1 記載此處轉場),**黑幕期間看不到任何內容,
無法判斷是否有「離開走位」被藏在淡出過程裡看不見**。下一幕(t≈204.5s 起)已經是完全不同的背景圭
(藍綠針葉密林,map31,對應 doc46「後山密林」),不是同一張草地圖的延伸鏡頭。

**結論**:「帶他離開」在本幕**沒有可驗證的視覺畫面**,合理解讀是:**用淡出/場景切換本身代表「兩人
一起離開」這件事**,不是靠额外一段可見走位表現。這與 doc46 §1「至少 2 處角色跨幀走位」的既有記錄
一致(該篇記錄的兩處跨幀走位是「索爾退朝」與「後山密林索爾+亞雷斯靠近悠妮蓋亞」,**不包含**
本篇新找到的「草地幕内亞雷斯接近+索爾接近」這兩段——本篇是該時間軸的**補篇**,發現了 doc46
當時沒有展開量測的兩段幕内走位)。

## 6. 教訓記錄(方法論,供類似量測任務參考)

**鄰域搜尋會被快速移動的角色甩開、鎖死在地形雜訊上**:第一輪嘗試對每幀用「上一幀位置 ±130px
鄰域」做模板搜尋更新位置,結果亞雷斯真實移動了 4.7 格後,追蹤器因搜尋窗口太小跟丟,鎖死在某個
地形明暗patch上(該處恰好與模板有中等相關,`v≈0.90` 看起來像「高信心」但其實是假鎖定,連續
70 秒座標紋丝不動,一度誤判為「亞雷斯全程沒動」)。改成**每幀對整個可視地圖區做全域搜尋**
(不依賴前一幀位置)後才拿到正確、乾淨的單調走位曲線。**信心值(NCC 相關係數)本身不足以識破
假鎖定——連續多幀「同一像素座標、同樣的中高信心值」反而是可疑訊號,應該對照原始截圖肉眼複驗**,
這正是本篇能推翻自己第一輪錯誤結論(「兩人對話全程零位移」)的關鍵回頭校正。

## 7. 第三輪複核:王座前終點(修正 doc48 §7)+ 草地會合終點再確認(2026-07-05)

> team-lead 指派:使用者實測發現兩個走位終點「差一點點」,需要精確量兩個終點。同樣模板匹配,
> 不目視;純分析不改碼。**發現一個貫穿全篇的系統性校準偏差,兩個終點都要用這個修正過的結果。**

### 8.1 校準偏差發現(兩個終點共用的方法論修正)

用王座廳的國王(fig48,FDFIELD 已知真值 `(7,5)`)/王后(fig66,已知真值 `(10,5)`)當「絕對真值錨點」
反查我的模板匹配 bbox-center 讀數,發現**兩者的 col 讀數都系統性偏高約 +0.43~+0.46 格**
(國王測得 7.43、王后測得 10.46,row 幾乎不偏,只差 +0.07~+0.11 屬雜訊範圍)。回頭比對 §1 表中
草地幕索爾起點的讀數(t=128.0 測得 (4.53,46.18),真值 (4,46)),**col 殘差 +0.53,同方向同量級**——
證實這不是單一場景的巧合,是本篇模板匹配方法**共用的系統性 col 偏差**(bbox 中心到 tile 錨點的
換算有約半格常數誤差,推測是 viewport 裁切偏移或 pitch 換算尾差)。**結論:所有本篇 col 讀數應
再減去約 0.45~0.5 格才是真值;row 讀數不必修正。** 因為修正量對同一幀內兩個角色是共用常數,
**角色間的相對位置(誰在誰哪一側、差幾格)不受此偏差影響,只有絕對格號需要修正**。
