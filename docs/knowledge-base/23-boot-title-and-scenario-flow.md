# 23 — 開機流程、標題動畫、主選單與劇情自動過場(反組譯)

> 本篇回答兩個一直沒系統性反組譯的問題:
> **(A)** 開機的「標題旋轉動畫 → FLAME DRAGON 標題 → 主選單」怎麼運作?
> **(B)** 新遊戲開頭「與前一代主角對話 → 自動記入(進入)戰場」這條自動過場怎麼串起來?
>
> 方法:以**靜態反組譯為主**(規則 62:控制流不是 runtime-only,先靜態追),
> 關鍵畫面再用**「已破解解碼器 + 解出輸出」當 oracle 視覺驗證**(規則 64),不依賴 DOSBox debugger。
> 工具:`tools/disasm_le.py`(本輪新增,capstone 反組譯 LE)+ `tools/le_xref.py`(fixup xref)。
> 位址皆為 **linear**(code 段 base = 0x10000)。每條結論標 **[驗]** 已驗證 / **[推]** 推定 / **[阻]** 受阻。

## 0. 先修正三個舊假設

追這條流程時,順手更正了先前 KB / 直覺裡的錯誤錨點:

| 舊假設 | 實際(已驗證) |
|---|---|
| `main = 0x10000` | **0x10000–0f 是 padding(int3/jmp/nop);0x10010 是「載入一關場景」子程式**(被 0x1a251、0x26130 呼叫),不是 main。 |
| 程式主函式在 code 開頭 | **真正的 main = `0x25bf4`**:無任何 caller、開頭做一次性音效/檔案子系統 init(0x379ee/0x3a722/0x38d3b…),是 Watcom CRT 直接進入的 main。 |
| 標題 logo = FDOTHER 資源 #0x65 | **#0x65 是 768B 調色盤(淡入用)**;FLAME DRAGON logo 在 **FDOTHER 資源 #7(巢狀 LLLLLL 容器)的 sub0**(320×200)。 |

## 1. 頂層狀態機骨架

沒有單一 `while(1)`;頂層是 **main(0x25bf4)內的雙層迴圈**,配合「離開即 `jmp 0x16f04` 回標題模組」構成 module 互跳的狀態機。**核心驅動變數只有一個:`[0x53c03]` = 目前章節**。

```
DOS/4GW entry 0x3c964 ──► Watcom CRT ──► main 0x25bf4
   │  一次性 init(AIL 音效 / 檔案 / 預載 FDOTHER+FDTXT 字模圖集)
   │
   ├─ play_bgm(track 0x12)                       ; 標題/主選單 BGM
   ├─ call 0x25ebb ── 標題序列 + 新遊戲/讀檔分流
   │      ├ choice = title_seq 0x1f894           ; 開機動畫+標題+主選單(見 §2/§3)
   │      ├ choice==0 新遊戲: [0x53c03]=0
   │      ├ choice==1 讀檔:   存檔槽選單 0x30550 → [0x53c03]=存檔章節
   │      └ call [ [0x53c03]*4 + 0x51d71 ]        ; 章節「戰前/劇情」跳表(見 §4)
   │
   └─ 戰役迴圈 ◄─────────────────────────┐
        ├ 戰場初始化 + call 0x117e7 戰場指令迴圈(逐單位行動)
        ├ [0x53ecc]==1 → 戰鬥事件 0x22e5c
        ├ [0x53ecc]==2 → call [ [0x53c03]*4 + 0x51de9 ] 戰後/勝利劇情 + 設下一章
        ├ 續打 → 回戰場初始化
        └ 結束 / quit → jmp 0x16f04(回標題/選單模組 0x16f55,選擇變數 [0x53c57])
```

**狀態變數(都在 obj2 0x50000 區)** [驗]：

| 變數 | 含意 | 寫入點 | 讀取/調度 |
|---|---|---|---|
| `[0x53c03]` | **目前章節(核心 game-state)** | 新遊戲=0(0x25ed6)、讀檔=存檔值(0x10147/0x26067)、戰後+1(0x25855/0x22f2c)、cutscene 暫借(0x32326) | 章節跳表 0x51d71/0x51de9、BGM 表 0x51e63、戰場公式 0x10b9e、章節文本 0x1088d |
| `[0x53c57]` | 選單目前選擇 index(各選單共用,進選單即歸 0) | 0x16f83/0x172ba/0x2600c | `cmp [0x53c57],0/1/2/3` 串接分派 |
| `[0x53ecc]` | 章節事件 handler↔戰役迴圈的 pending 碼(1=進世界地圖/中場、2=勝利/章節結束、0=閒置) | 章節事件 handler(0x51b19→0x205xx,28 處設 1/2,見 doc 25/26);戰役迴圈清 0 | 戰役迴圈 0x25dd5/0x25e02;過場讀==0。完整狀態機見 doc 24 §6 |

## 2. 標題序列(A 之一):角色立繪「垂直捲動」進場

`title_seq 0x1f894`(唯一 caller = 0x25ec8)是整個開機畫面本體,三階段:**捲動動畫 → 抹除轉場 → 主選單**。

### 2.1 「旋轉動畫」真相 = 預存 5 幀直疊 + 視窗垂直捲動 [驗]

不是即時旋轉、不是逐像素 rotate、不是調色盤循環。實作是:

1. **組大圖**(0x1fa1c–0x1fa55):載入 FDOTHER 資源 **#0x45–0x49 共 5 幀**,每幀 **320×147**(檔頭 `40 01 93 00` = w=0x140,h=0x93 鐵證),用 sprite blit(0x4e63d)依序疊到一個 **320×735** 大緩衝(各幀 y=0/147/294/441/588)。
   ```
   0x1fa1e  lea eax,[esi+0x45]          ; 資源索引 0x45..0x49
   0x1fa26  push 0x1a4d                 ; ->"FDOTHER.DAT"
   0x1fa2b  call 0x111ba                ; 載入
   0x1fa3f  imul ebx,esi,0x93           ; y = 147*frame
   0x1fa49  call 0x4e63d                ; blit 到大緩衝
   ```
2. **垂直捲動**(0x1fa85–0x1fc64):視窗高度 200,來源列 `esi` 從 **0x217(535)遞減到 0**,每步把「大緩衝 + esi*320」起的 320×200 複製到 VGA 0xA0000。即畫面從立繪底部往上掃到頂部。
   ```
   0x1fa85  mov esi,0x217
   0x1faae  add eax,edi ; src = 緩衝 + esi*320
   0x1fabb  call 0x11eb0 ; 複製 320×200 視窗 → 0xa0000
   0x1fc59  call 0x10620 ; 偵測按鍵(讀 BIOS kbd head[0x41a]/tail[0x41c])→ 可跳過
   ```
   配合 palette fade-in(0x1f525)做淡入。任意按鍵可跳過(0x10620 偵測、0x4e031 清鍵盤緩衝)。

### 2.2 視覺驗證(規則 64:解碼器當 oracle)[驗]

把 #0x45–0x49 用已破解的 **FIGANI 4-mode sprite RLE**(`0xff`=mode11 透明,吻合「sprite blit + 透明」)解出、直疊成 320×735:
→ **是一尊藍紅色調的炎龍/魔王肌肉立繪**(惡魔臉、披風、胸前紋路)。本機 `extracted/title_re/title_scroll_big.png`。
所以玩家看到的「動畫」= 這張高立繪由下往上捲過螢幕,觀感上像角色緩緩升起/逼近。**「旋轉」一說在反組譯與解圖兩邊都不成立,真相是垂直捲動。**

### 2.3 FLAME DRAGON 標題如何「打上去」[驗]

捲到立繪頂端後,經 BG.DAT 抹除轉場(0x286bd,esi=0x28→0),載入**主選單畫面 = FDOTHER 資源 #7**(巢狀 LLLLLL 容器,8 子資源):
- **sub0(320×200)= 標題主畫面**:解出可見大大的 **「2」** + 華麗龍紋 **FLAME DRAGON** logo + 卷軸副標 **「LEGEND OF GOLDEN CASTLE」**(炎龍騎士團2 黃金城傳說)。本機 `extracted/title_re/sub7/s00_spr.png`。
- **sub1–6 = 3 個選單項 ×2 態**(61×7 / 62×8,「未選 / 反白選中」),如第一項 "START"。
顯示方式 = 直接 blit 預存圖 + palette fade-in,**非飛入/縮放**。

> ⚠ **本節部分結論已被 §2.4(第13輪 dosbox 實機觀察)推翻/修正**:上面「非飛入/縮放」只對「主選單畫面本身的靜態合成」成立,但**logo 進場那一瞬間確實有縮放動畫**(見 §2.4①);且本節與 §2.1 描述的「捲動」只是整段開場動畫裡的一小段,前面還有一大段未記錄的敘事過場(§2.4③)。靜態 RE(FDOTHER #0x45-49/#7)本身沒錯,錯的是「這就是玩家看到的全部開場」這個推論範圍。

### ① logo 進場方式:**裁決 = 純縮放(zoom),不是旋轉**,且 doc23 §2.3「非飛入/縮放」的結論不成立

實機看到的是「**紅色閃光轉場 → 巨大特寫的『2』數字用約 0.5 秒快速縮小定位 → 白色閃光 → 色板淡入到最終畫面**」,不是捲動揭露,也不是逐像素旋轉:

**結論**:
- **只有「2」這個數字做了明顯的縮放動畫**(從遠大於最終尺寸的特寫快速縮小定位),**FLAME DRAGON 文字本身沒有隨之縮放**,是在「2」縮小過程中以固定最終尺寸「顯像」進來的(fade/wipe,非 scale)。
- 縮放期間**看不到透視變形/傾斜/翻轉**(「2」的鉤爪筆畫形狀在 188→191 全程一致),所以**視覺證據支持「縮放(zoom)」,不支持「旋轉(rotate)」**。使用者記憶的「由螢幕外旋轉縮到螢幕內」——「由外而內縮小」的部分**證實成立**,但「旋轉」的部分**這次沒有拍到證據**,如實回報而非附和。
- 全程從紅閃光到選單靜止只有 **~1.4 秒**(184→194,見上表),非常快、偏「一擊到位」的衝擊感,不是緩慢優雅的飄入。
- 這推翻 doc23 §2.3「顯示方式 = 直接 blit 預存圖 + palette fade-in,非飛入/縮放」——**logo 出場那一刻確定有縮放動畫**,只是它可能是「~3-6 張預先算好不同尺寸的『2』點陣圖快速依序 blit」(類似 §2.1 已驗證的「預存多幀直疊」手法),不必是 runtime 即時縮放運算;這點留給下一輪靜態 RE 去 FDOTHER/ANI.DAT 找對應的多尺寸「2」資源驗證,本輪只裁決「有沒有縮放」「是不是旋轉」這兩個視覺問題。

### ⑤ 轉場(捲動→標題)長什麼樣

不是漸變淡出淡入的柔和轉場,是**兩段硬切閃光**:故事過場最後一幕(紅光惡魔臉特寫)**直接切成全螢幕純紅色**(frame_185–186,無漸變過程,一幀之內從有內容變全紅),隔一幀黑幀後「2」開始縮放進場(見①),縮放定位完成後**再一次全螢幕泛白閃光**(frame_192,bloom 效果),然後色板才收斂到最終橘紅漸層畫面。兩次閃光(紅/白)夾住 logo 縮放動作,是典型「衝擊感」轉場手法,不是 doc23 §2.1 描述的「BG.DAT 抹除轉場」那種漸進式擦除——**抹除轉場可能用在別的段落轉換,不是這裡**(本輪未逐一比對每個分鏡間的轉場手法是否都一致,只確認了「捲動→標題」這一段是閃光式)。

### ⑥ `0x1f73f`:FDOTHER 靜態全螢幕幕(第14輪反組譯,補 doc39 §10.7 待補項)[驗]

> 承接 doc39 §10.5/§10.7:title_seq 捲動迴圈內 `esi==0x1c2`(450)與 `esi==0xa`(10)各觸發一次
> `call 0x1f73f(esi,edi,idxA,idxB)`,是**獨立於 ANI.DAT/AFM VM 的第二套顯示機制**——不播動畫,
> 是「黑幕 → 全螢幕靜態圖 blit → 調色盤淡入 → 停留 → 復原捲動畫面」的一次性插播。純靜態反組譯
> (規則 62)逐指令 esp-relative 反推(`push N; call 0x36cd7` 經查證是 Watcom 堆疊 guard-page 探測,
> **對 esp 淨效果為 0**,不是配置區域變數——與 doc39 §10 的 esp_rel 校正同一教訓,別假設常見慣例)。

**`0x1f73f` 完整流程**(cdecl,呼叫端 push 順序 `esi,edi,idxA,idxB` 反推出形參序
`(idxB, idxA, edi, esi)`,對應 `[esp+4]..[esp+0x10]`):

```
1. memset(0xA0000, 0, 0xFA00)                          ; 清 VGA 畫面(320×200=64000B 全黑)
2. load("FDOTHER.DAT", oldbuf=[cache 0x53a65], idxA)    ; idxA → 覆寫進既有快取(調色盤,見下)
3. load("FDOTHER.DAT", oldbuf=0, idxB)                  ; idxB → 全新配置(全螢幕圖,見下)
4. blit(eax=idxB圖指標, 0,0, 0xA0000, 0x140, -1)         ; 0x4e63d 全螢幕 blit(320×200)
5. call 0x1f525                                          ; 調色盤淡入
6. wait_ticks(1); wait_ticks(6)                          ; ⚠0x17aa9 是 tick 計數忙等(讀全域計數器
                                                            [0x46c],非 doc39 §10.7 原猜測的 BGM/SFX——
                                                            本輪修正)
7. call 0x1f882                                          ; 同步/vsync 輔助(語意同 doc39 §10.7 未展開)
8. load("FDOTHER.DAT", oldbuf=[cache], idx=0x65)         ; 重載 #0x65(標準 768B 淡入調色盤)進快取
9. addr = edi + esi*320                                  ; edi/esi = title_seq 傳入的捲動緩衝基址/列號
10. copy_rect(dst=0xA0000, dst_w=0x140, src=addr, 0x140, 0x140, 0xc8)  ; 0x11eb0,320×200
    → 復原插播前 title_seq 自己正在捲動的那一幀
11. call 0x1f525                                          ; 再次調色盤淡入
```

**idxA/idxB 語意查證**(讀 `extracted/raw/FDOTHER/` 實體 header 確認,非猜測):

| 觸發 esi | idxA(載入快取) | idxB(全螢幕 blit) | idxA 檔案 header | idxB 檔案 header |
|---|---|---|---|---|
| `0x1c2`(450) | `0x63`(99) | `0x64`(100) | `FDOTHER_099.bin` 768B(**調色盤**,無 w/h 頭) | `FDOTHER_100.bin` header `40 01 c8 00`=320×200 |
| `0xa`(10) | `0x4c`(76) | `0x4b`(75) | `FDOTHER_076.bin` 768B(**調色盤**) | `FDOTHER_075.bin` header `40 01 c8 00`=320×200 |

→ **idxA 一律是 idxB 的專屬 768B 調色盤**(同 #0x65 的 6-bit RGB 格式),不是另一張圖——doc39 §10.4
早先把 `#0x63`(99)註記為「紅底背景」是描述它套用後的視覺效果(調色盤偏紅),不是獨立圖層,本節校正
用詞但不推翻結論。

**視覺比對(規則 64:已破解 sprite RLE 解碼器 `decode_sprite.py` 當 oracle,配對應調色盤解出)**:

- `idxB=0x64/#100` + `idxA=0x63/#99` 調色盤 → 解出**紅底盔甲守護者半身像**(`extracted/title_re/fdother_static/fdother100_guardian_pal099.png`,本機)。比對 dosbox `frame_012–014/027–028`(§2.4③ item①「守護者盔甲全身像細節」),構圖/色調一致。**結論:`esi==0x1c2` 這幕 = item①守護者的靜態收尾畫面,不是 doc39 §10.7 原猜測的「分鏡⑨惡魔臉」**。
- `idxB=0x4b/#75` + `idxA=0x4c/#76` 調色盤 → 解出**滿月前浮空城剪影**(`extracted/title_re/fdother_static/fdother075_floating_castle_pal076.png`,本機)。與 dosbox `frame_168–173`(§2.4③ item⑥「浮空城剪影疊在滿月前」)**逐像素構圖/配色吻合**(月面斑紋位置、城堡剪影輪廓、下方水波紋理皆對上)。**結論:`esi==0xa` 這幕 = item⑥浮空城,確認**。

**修正 doc39 §10.7 待補項**:`0x1f73f` 只在 title_seq 內被呼叫兩次(全域 `calls 0x1f73f` 僅命中
`0x1fbaf` 一處呼叫指令,兩個觸發點共用同一條 `call`,同 §10.5 wrapper 模式),對應**恰好兩幕**靜態圖

## 3. 主選單機制(A 之二)[驗]

輸入迴圈 0x1fe2c–0x1feeb,讀鍵函式 0x36d98 填 `[0x3a8d]=ASCII`、`[0x3a8e]=scancode`:

| 按鍵 | scancode | 動作 |
|---|---|---|
| ↑ | 0x48 | 游標 `ebp`−−,到 0 則 wrap 到底;播 SFX 0x25a96 |
| ↓ | 0x50 | 游標 `ebp`++,到頂則 wrap 到 0;播 SFX |
| Enter / Space / 0xe0 / 0x52 | 0x0d/0x20 | 確認(選中項閃爍 4 次 → 回傳 `ebp`) |

- 游標索引 = `ebp`,選單項由 0x1ff79 繪製/反白;項數與預設項來自設定結構 0x5204e。
- **此主選單迴圈未見 ESC 分支**(只有移動 + 確認)[驗];ESC 取消是戰場/子選單的行為(見 doc 13),非主選單。
- 回傳值分派(回 0x25ebb):**0 = 新遊戲**(→ §4)、**1 = 讀檔**(存檔槽選單 0x30550,3×64000B 全螢幕 VGA 快照緩衝)、其他 = 安靜載入。

## 4. 新遊戲 → 開場對話 → 自動進戰場(B)

這條就是使用者說的「跟前一代主角對話,然後自動記入戰場」。**全程由 `[0x53c03]` 一個變數驅動,中間沒有任何「要不要進戰場」的玩家選擇點 → 即腳本化自動過場。**

```
選「新遊戲」(0x25ebb)
   ├ [0x53c03] = 0                                   ; 章節歸零
   └ call [ 0*4 + 0x51d71 ] = 0x3231b               ; 章節0「戰前/劇情」處理器
        │
章節0 處理器 0x3231b = 開場序章 cutscene  ←★「與前代主角對話」在此
   ├ [0x53c03]=0x20; call 0x205da                    ; 暫借章節值切到 FDTXT 序幕「影像」資源
   ├ 0x15f84 ×N + 0x13185 淡入                        ; 逐格貼開場畫面
   ├ play_bgm(0x0b)                                   ; 序章配樂
   ├ [0x53c03]=0x1f; call 0x205da                     ; 切到 FDTXT 序幕「對話」資源
   ├ push 0x5a..0x69; call 0x1366a (逐段)             ;★ 逐段播對白(渡海/海盜/哈瓦特父子等 beats)
   │      └ 對白走文本渲染器 0x16D00 區;控制碼 0xFFEF 依說話者 ID 從 DATO.DAT 載入頭像(doc 14)
   ├ call 0x10b4e (載入登場單位 roster)
   └ ret                                              ; cutscene 結束 → 回 driver 0x25ebb 內續行(非相位機,見 doc 24)
        │
driver 0x25ebb 內續行 → 進入戰場設置 0x10010(0x26130)
   └ 戰場地圖編號 = [0x53c03]*3 + 2   (0x10b9e)       ;★ 不是寫死常數,由章節推導
        0x10b9e  mov edx,[0x53c03]                    ; 章節
        0x10ba6  shl eax,2; sub eax,edx; add eax,2    ; eax = 章節*3 + 2
        0x10bb5  push 0x1a59 (="FDFIELD.DAT"); call 0x111ba
   → 載入我方隊伍(每單位 0x50B 結構)+ 敵方(0x10b4e)+ 佈署 → 戰鬥迴圈 0x117e7
```

要點:
- **章節文本載入器 0x1088d** 取 FDTXT 資源 `章節+1`;新遊戲章節=0 → 資源 1 = 序章。[驗]
- **FDFIELD 每張地圖佔 3 個資源**(構成 / 控制+寶箱 / 出場位置,見 doc 03/91);`章節*3+2` 取其一,第一場 = 章節0 對應資源組。[驗]
- **「自動」的本質**:`[0x53c03]` 同時決定「播哪段對話 / 載哪張戰場 / 放哪首 BGM / 呼叫哪個章節腳本」。
- **cutscene→戰場是同一 driver(0x25ebb)內的線性串接**(doc 24 用 call-graph 釘死):driver 先 `call [章節*4+0x51d71]`(章節0=cutscene 0x3231b),handler `ret` 後 driver 繼續走到 `0x26130 call 0x10010` 進戰場;**中間無玩家選擇點 = 自動過場**。這正是原版「33 段固定流程」的硬編碼版本。[驗]
  > 修正:早先描述用「相位機接手」串接 cutscene→戰場,經 call-graph 反向追證實是**同 driver 內線性執行**;相位變數 `[0x53ecc]` 其實是**戰後**分支(1=事件、2=勝利→跳表 0x51de9),非 cutscene→戰場的中介。詳見 doc 24。

### 戰後流向 [部分驗]
戰鬥結束(`[0x53ecc]==2`)→ 章節「戰後」跳表 `0x51de9[章節]`(ch0=0x22ef6:貼結束畫面 → `[0x53c03]=1`)→ 相位 `[0x53ecc]==1` 可進世界地圖/中場(0x22e5c)→ 下一章戰前跳表。完整勝/敗分歧未逐條追完。

## 6. 對重製(Go/Ebiten)的意義

這條反組譯**直接定義了 ScenarioRunner 的原版行為**,可一對一落地(見 doc 19/21):

- **章節驅動 = 線性 campaign.json**:原版 `[0x53c03]` 章節變數 → 我們的 `currentNode`;章節跳表 0x51d71/0x51de9 → 每節點的 `on_enter`(劇情/cutscene)與 `on_win`(戰後/下一節)。
- **戰場地圖 = 章節*3+2**:自動產生線性 campaign 時,第 N 章戰場 = FDFIELD 資源組 `[N*3, N*3+1, N*3+2]`,可機械生成(更新 `tools/export_engine_assets.py` 的選圖邏輯)。
- **開機標題**:重製可忠實重現「立繪垂直捲動 + FLAME DRAGON logo + 主選單」,素材即 FDOTHER #0x45–0x49 與 #7(玩家自備原版產生,不入庫)。
- **自動過場**:cutscene→戰場無玩家選擇點的串接,對應 ScenarioRunner 的 `auto` 轉場(無 choice 節點)。
