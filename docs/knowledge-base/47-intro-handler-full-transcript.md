# 47 — 序章 handler 0x3231b 完整轉錄(過場腳本機制的第一性原理解答)

> 回答使用者本質問題:「對話場景轉換/戰場觸發應該都在原版腳本中,RE 為什麼沒還原?」
> 結論:**FD2 的過場=三層架構**——文本(FDTXT 資料)+ 單位演出(acting 資源,由 0x1366a 播放)
> + **編排順序(EXE 每章一支寫死的 handler)**。先前只還原了第一層與原語位址,
> 編排層從未逐 beat 轉錄 → remake 一直靠影片/記憶手工近似,所以「局部像、整體不像」。
> 本篇 = 序章 handler 的逐 beat 轉錄(方案 a),同時證偽「staging 藏在 FDTXT 控制碼」(方案 b)。

## 1. 方案 b 證偽:FDTXT 沒有 staging 控制碼

掃描與序章相關的全部文本資源,`>0x71F` 值直方圖:

| 資源 | 字串數 | 出現的控制碼 | 未知碼 |
|---|---|---|---|
| FDTXT_032(密林幕文本) | 11 | FFEC/ED/EE/FD/FE/FF | **無** |
| FDTXT_033(王城幕文本) | 6 | FFEC/ED/EE/FD/FE/FF | **無** |
| FDTXT_002(第一章戰鬥文本) | 17 | 同上 | **無** |

檔頭=字串 offset 表(FDTXT_001 前 12 個 u16 遞增值即是)。**文本流純對話,無走位/鏡頭指令。**

## 4. 對 remake 的直接修正指示

1. **王城→草地轉場改「同 map32 鏡頭平移」**(beat 1-8):現行 remake 是淡出換節點,原版是 `0x135dd(0,43)` 直接滑下去。密林幕才是真的換圖(+palette fade)。
2. 對白與演出**交錯**的節奏(txt#N 與 acting#M 一一穿插)與現行「一幕一段」不同——每條 txt 字串前後常有一段演出,remake scene 內要支援「對白←→走位」交替(部分已有:WalkFirst/storyWalks,需擴充 per-line 演出)。
3. 海島幕鏡頭有三個平移點((4,12)→(0,0)→(0,15)),不是固定一鏡。
4. 入隊時點=密林幕結束後、海島幕之前(0x112a5×4)。
5. 草地幕配樂從 track 11 開始(0x25977(0xb));王城幕開頭 bgm 是停止狀態。

## 5. 未解(誠實列表,下一輪 RE 錨點)

- **0x627d8 acting 資源表的填表點**:refs 只有 getter 0x4e803;已排除=章節載入器 0x1088d/0x205da、
  0x111ba(純 fopen/fseek/fread)、0x10652(章節大圖)。表在 BSS,填表用暫存器基底寫入,靜態難尋。
- **acting 資源所在容器**:格式掃描已排除 FDTXT(35)/FDFIELD(100,roster 無尾掛)/DATO(137,90-136 全頭像量級)
  /ANI(第10個=0 bytes)/EXE 資料段(無吻合 blob)。候選:FDOTHER 巢狀子資源、FIGANI 尾段、FD2.TMP。
  或 id 有 base 偏移。**建議下一輪:dosbox 記憶體 dump [0x627d8] 表內容反查來源**(規則 64 第三條路)。
- acting 幀的完整語意:0x80 旗標分支、unit+3/+4 欄位如何被 0x11cac 用於中間位置繪製(=走位內插的實作)、
  鏡頭是否隨 acting 內容移動(影片顯示走入時鏡頭跟隨,但 handler 只在 beat 間平移——疑 0x11cac 追焦或 acting 資源含鏡頭指令)。
- `[0x53afb]` 語意;`0x32975/0x32999` 內部。

## 6. 方法紀錄

b(便宜假說)→掃直方圖 30 分鐘證偽→a(handler 轉錄)。原語身分靠「參數模式+資料側交叉驗證」
(txt 條數 6/11/12 對上 0x15f84 索引範圍;play_bgm 位址對上記憶;0x112a5 四個參數=四主角 id)。
教訓同 62/64:**先資料側驗證假說再深挖 code**,兩處舊誤判(0x15f84「逐格貼圖」、「無行軍動畫」)都是
「機制懂了但內容沒轉錄」時代的過度推論。
