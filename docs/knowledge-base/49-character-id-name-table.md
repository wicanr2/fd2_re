# 49 — portrait/char id → 角色名總表(對話反推,第 N 輪)

> 任務起點:worklist 寫「140 組 portrait→角色名,已知 12 個」。本輪核實後這個起點數字本身就已過期
> ——`docs/data/exe_tables/characters.json`(0-31,32 筆)+ `tools/export_units.py` 的
> `PORTRAIT_CLS_NAME`(68/96/97)+ 先前 RE 對圖確認的國王(48)/王后(66),**光是已定案的就有 38 組**,
> 不是 12 組。本輪在既有基礎上新增 **1 組**(id 126 = ASR-06),並用獨立方法交叉驗證既有結論。
> 完整結果見 `docs/data/portrait_names.json`。

## 一句話總覽

用**對話 identity-tag 反推**(非單純讀人名字串,見下)把 `remake/assets/story/ch*.json`(35 個檔,
1533 句已人工精校對白)對回 raw `FDTXT_NNN.bin`,抽出每句話真正的控制碼與 operand,只採信
`-17`/`-18`(身分標籤,經 `0x12C60` 查表,同一角色跨章節穩定)這條路徑;`-19`/`-20`(場景單位陣列
index)按 `40-speaker-portrait-mapping.md` 已證實的結論,**同一數字在不同場景可能指不同人,不可
單獨採信**。全數 0-31 既有結論獲得交叉驗證(無一衝突),另確認 id 126 = 最終魔王 ASR-06。

## 方法(對話反推 SOP)

1. **抓穩定的原始碼,不是抓渲染後的名字字串**:`tools/decode_story_text.py` 產出的
   `full_story_auto.md` 對 id>31 的說話者會把 operand 誤當字模解碼,洩漏成單一亂碼字元(與
   `render_story.py` 同源 bug,`40` 已定論)。本輪改寫獨立解碼(`decode_turns`,見下),正確消耗
   `-17..-20` 後的 operand word,不讓它落進文字流。
2. **對齊**:`remake/assets/story/ch*.json` 是已人工精校的正確譯文(35 章,1533 句,含所有 NPC 的
   正確中文名),但其 `speaker` 欄位只查 `characters.json`(0-31),查不到一律 `-1`——即使 raw 資料
   其實有 operand 數字也沒被記錄。本輪用 `difflib` 對每章的「raw turns 文字序列」與「json 行文字
   序列」做正規化(僅留中文/英數)後的最長公共子序列對齊,一句句把 json 的 `speaker_name` 對回其
   raw `(control_code, operand)`。1533 句中對齊成功 1417 句(92%),其餘因人工譯文標點/分段與原文
   略有出入而對不上,不影響下列結論(結論皆來自數十次重複、跨多章的穩定值)。
3. **只採信 `-17`/`-18`**:每個具名角色的 `-17`/`-18` operand 高度一致(如亞雷斯 106 句
   `-18,4` + 6 句 `-17,4`,無一次落在其他數字);`-19`/`-20` 則同一角色在不同章給出不同數字
   (如萊汀入隊前 `-19,11`、入隊後穩定 `-18,6`——**同一人,前者是「目前場景單位陣列第 11 格」,
   後者才是真正身分標籤 6**,與 `40` 文件 ch09 案例完全吻合,這裡是第二個獨立重現此機制的案例)。

## 為何不是「140 組全部可由對話反推」——重要的方法論修正

本輪起手前的任務假設是「speaker id = portrait id = 單位 char id(同一編號空間),逐句掃描即可批量
還原」。實測下來這個假設**只對兩種情況成立**:

1. **32 個玩家角色**(0-31):有永久身分標籤,`-17/-18` 全程穩定。
2. **少數反覆出現、開發者刻意賦予固定身分標籤的 NPC/怪物 class**(如 48/66/68/96/97/126)。

**對絕大多數其餘 id(估計 90+ 組)不成立**:1533 句對白裡,除上述已知角色外的其餘對白
(海盜甲/乙/丙/丁、強盜 B/C/L/M/N、獸人甲/乙/丙/丁……)**清一色走 `-19/-20`(場景單位陣列
index)**,而非 `-17/-18`。`40-speaker-portrait-mapping.md` 已證實這條路徑的數字**隨地圖/場景
重新填入,同一數字在不同章可能指向完全不同的單位**——這正是本輪重新驗證到的萊汀入隊前後案例。
換言之,這些泛用敵人/路人 NPC 對話裡的數字**不是**它們的全域 DATO portrait id,若強行採信會把
「這場戰鬥剛好站在第 11 格的某個獸人」誤記成一個固定的角色。

要繼續往下補,唯一可靠的路是 `tools/export_units.py` 已驗證的方法:**逐張地圖解 FDFIELD 出場位置
段的 `byte[+7]`**,拿實際場景 roster 與 `DATO_NNN` 頭像圖交叉核對(48/66/68/96/97 皆用此法定案)。
這是比對話反推大得多的工程(每張地圖都要解),不是本輪範圍,列入 worklist 待辦。

## 發現的資料品質問題(附帶,非本輪任務,但應記錄避免誤用)

`remake/assets/story/` 部分章節 JSON(至少 `ch05/08/14/18/21/22/25/27/28`)的 `speaker_name`
欄位中,混有單一亂碼漢字(如「惡」「箭」「聖」「武」「弓」「狙」「侶」「大」「約」等)——這是
**與 `40` 文件已定論的同一種洩漏 bug**(某段生成管線把 operand 數字誤丟進 `glyph_map` 解碼當文字
用),**不是**真正的說話者名字。`ch25.json` 已有一筆人工加註(`"note": "[?] 原始代碼顯示為
H(米亞斯多德)…已合併歸屬"`)顯示先前已有人發現此問題並手動修正,但其餘實例(數十筆)尚未清理。
本輪未展開清理(超出「建表」範圍),僅記錄示警:**引用這些章節的 `speaker_name` 前,先用本文件
的 `decode_turns` 方法或肉眼核對原文,不要直接信任單字漢字說話者名**。
